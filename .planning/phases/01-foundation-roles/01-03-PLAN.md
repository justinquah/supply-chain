<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client component → server action | Untrusted form input (email, role, userId) crosses into privileged invite/role-change |
| server action → Supabase Admin API | Service-role client bypasses RLS; must stay server-only |
| browser → /login | Unauthenticated request; must not 500 on missing config |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | Information Disclosure | SUPABASE_SERVICE_ROLE_KEY leaking into the client bundle | mitigate | admin.ts is server-only (no "use client", no NEXT_PUBLIC_ prefix) and imported only from the "use server" actions; users-table.tsx must not import @/lib/supabase/admin (acceptance criteria assert count 0) |
| T-01-07 | Elevation of Privilege | Crafted role value in invite/updateRole escalates a user | mitigate | requireRole("ADMIN") guards both actions; role validated server-side against appRoleSchema (four roles only) before any DB/admin write |
| T-01-08 | Denial of Service | Missing Supabase env var crashes /login with a 500 | mitigate | createClient() in client.ts/server.ts throws a clear configuration error before calling the Supabase factory; human-check confirms production /login renders |
| T-01-09 | Spoofing | Stale role in JWT vs profiles table | accept | getCurrentUser() always reads role from the profiles table, not JWT claims (already correct in the codebase) |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan (removal-only phase) |
</threat_model>
